Phase 1

Foundations — Threads, Lifecycle & Creation

Before writing a single synchronized or volatile, you need to know exactly what a thread is, how the JVM maps it to the OS, what states it goes through, and all the ways to create one. This chapter is your bedrock.

1.1 What Is a Thread?

A process is an isolated program in memory — it has its own address space, file handles, and OS resources. A thread is the unit of execution inside a process. Threads inside the same process share heap memory, static fields, and open files — but each thread has its own stack, program counter, and local variables.

PropertyProcessThread
MemoryIsolated address spaceShared heap, own stack
Creation costHeavy (fork/exec)Lighter (~1 MB stack by default)
CommunicationIPC (pipes, sockets)Shared memory (be careful!)
Crash impactDoes not affect othersCan crash entire JVM

JVM Thread Model — Platform Threads

In Java, a platform thread (also called an OS thread) is a thin wrapper around a real OS thread. The JVM delegates scheduling entirely to the OS. Each platform thread consumes about 1 MB of native stack memory. This is why creating 10,000 platform threads will crash your JVM — but 10,000 virtual threads (Phase 11) is completely fine.

1.2 Thread Lifecycle

A Java thread moves through exactly six states defined in Thread.State:

StateWhat it means
NEWThread object created, start() not yet called
RUNNABLERunning on CPU or ready to run (waiting for CPU slice)
BLOCKEDWaiting to acquire a synchronized intrinsic lock
WAITINGWaiting indefinitely — via wait(), join(), park()
TIMED_WAITINGWaiting with a timeout — sleep(ms), wait(ms), join(ms)
TERMINATEDFinished — run() returned or threw an uncaught exception
A thread in RUNNABLE may not be on the CPU right now — it is eligible to run. The OS scheduler decides who actually runs. BLOCKED is specifically for synchronized lock waiting; lock waiting via ReentrantLock shows as WAITING.

1.3 Creating Threads — All 4 Ways

Way 1: Extend Thread

public class CounterThread extends Thread {
    private final String label;

    public CounterThread(String label) {
        super(label);          // set thread name
        this.label = label;
    }

    @Override
    public void run() {
        for (int i = 1; i <= 5; i++) {
            System.out.println(label + " → " + i);
        }
    }

    public static void main(String[] args) {
        CounterThread t1 = new CounterThread("A");
        CounterThread t2 = new CounterThread("B");
        t1.start();   // spawns an OS thread, calls run() on it
        t2.start();
        // Output order is non-deterministic — that is the point
    }
}
Never call t.run() directly — it executes on the current thread, no new thread is spawned. Always use t.start().

Way 2: Implement Runnable (Preferred over extending Thread)

public class PrintTask implements Runnable {
    private final String msg;

    public PrintTask(String msg) { this.msg = msg; }

    @Override
    public void run() {
        System.out.println(Thread.currentThread().getName() + ": " + msg);
    }
}

// Usage
Thread t = new Thread(new PrintTask("hello"), "worker-1");
t.start();
Prefer Runnable over extending Thread. Your task class stays free to extend something else, and the task is decoupled from how it runs.

Way 3: Callable + Future (when you need a return value)

import java.util.concurrent.*;

public class SumTask implements Callable<Long> {
    private final int n;
    public SumTask(int n) { this.n = n; }

    @Override
    public Long call() {
        long sum = 0;
        for (int i = 1; i <= n; i++) sum += i;
        return sum;
    }
}

public static void main(String[] args) throws Exception {
    ExecutorService exec = Executors.newSingleThreadExecutor();
    Future<Long> future = exec.submit(new SumTask(1_000_000));
    System.out.println("Result: " + future.get()); // blocks until done
    exec.shutdown();
}

Way 4: Lambda / Method Reference

// Runnable lambda
Thread t1 = new Thread(() -> System.out.println("hello from lambda"), "lambda-t");
t1.start();

// Thread.Builder API — Java 21+
Thread t2 = Thread.ofPlatform()
        .name("worker")
        .daemon(true)
        .start(() -> System.out.println("platform thread via builder"));

// Virtual thread (Java 21+)
Thread vt = Thread.ofVirtual()
        .name("vthread")
        .start(() -> System.out.println("virtual thread!"));
vt.join();

1.4 Thread Control Methods

sleep — pausing a thread

try {
    Thread.sleep(1000);                          // 1 second
    Thread.sleep(Duration.ofMillis(500));        // Java 19+ Duration overload
} catch (InterruptedException e) {
    Thread.currentThread().interrupt();           // restore interrupt flag!
}
When you catch InterruptedException, always call Thread.currentThread().interrupt() to restore the flag unless you are the top-level thread owner. Swallowing it silently breaks cooperative cancellation.

join — waiting for another thread

Thread worker = new Thread(() -> doWork(), "worker");
worker.start();
worker.join();                  // main thread waits until worker finishes
worker.join(5000);              // wait at most 5 seconds
System.out.println("worker done: " + !worker.isAlive());

Cooperative Cancellation via interrupt()

public class CancellableTask implements Runnable {
    @Override
    public void run() {
        while (!Thread.currentThread().isInterrupted()) {
            try {
                System.out.println("working...");
                Thread.sleep(200);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt(); // restore flag, then exit loop
                break;
            }
        }
        System.out.println("task cancelled cleanly");
    }

    public static void main(String[] args) throws InterruptedException {
        Thread t = new Thread(new CancellableTask());
        t.start();
        Thread.sleep(1000);
        t.interrupt();  // politely asks the thread to stop
    }
}

Daemon Threads

A daemon thread is a background thread that does not prevent the JVM from exiting. When all non-daemon threads finish, the JVM exits, and any remaining daemon threads are killed abruptly.

Thread logger = new Thread(() -> runForever(), "bg-logger");
logger.setDaemon(true);   // must be set BEFORE start()
logger.start();
Never use daemon threads for tasks that need to complete cleanly (writing to database, flushing buffers). They die without cleanup.
Phase 2

Java Memory Model — The Rules of the Game

The Java Memory Model (JMM) is the most misunderstood part of multithreading. It defines what values a thread is allowed to see when reading a shared variable. Without understanding the JMM, you cannot reason about any concurrent code — even code that "looks" thread-safe.

2.1 The Problem — Why Shared Memory Is Dangerous

Modern CPUs do not write directly to main memory. They write to L1/L2/L3 caches first. Each CPU core has its own cache. When Thread A writes a value on Core 1, Core 2 running Thread B may still see the old value in its cache. This is the visibility problem.

On top of that, compilers and CPUs are allowed to reorder instructions as long as the result looks correct from a single-threaded perspective. From another thread's perspective, this reordering can look catastrophic.

The Broken Flag — Visibility Failure in Action

public class BrokenFlag {
    private static boolean running = true;  // NOT volatile

    public static void main(String[] args) throws InterruptedException {
        Thread worker = new Thread(() -> {
            while (running) { /* spin */ }
            System.out.println("stopped");
        });
        worker.start();
        Thread.sleep(100);
        running = false;  // main thread writes this
        // worker may NEVER see this change — loops forever on some JVMs!
    }
}
This program may run forever on a server JVM with JIT enabled. The JIT can hoist running into a register, and the worker never re-reads from memory. This is legal under the JMM without volatile.

2.2 happens-before — The Formal Rule

The JMM defines a happens-before (HB) relationship. If action A happens-before action B, then A's writes are visible to B. The key HB rules:

RuleWhat it means
Program orderEach action in a thread HB any later action in the same thread
Monitor unlockUnlock of a monitor HB any subsequent lock of that same monitor
volatile writeA write to a volatile field HB any subsequent read of that field
Thread startt.start() HB any action in thread t
Thread joinAll actions in thread t HB t.join() returning
TransitivityIf A HB B and B HB C, then A HB C

2.3 volatile

volatile gives you two guarantees:

  • Visibility: a write to a volatile variable is immediately visible to all threads that subsequently read it
  • Ordering: reads and writes to volatile variables cannot be reordered with surrounding reads/writes

volatile does NOT give you atomicity for compound operations like i++.

Fixed Flag with volatile

public class FixedFlag {
    private static volatile boolean running = true; // volatile fixes it

    public static void main(String[] args) throws InterruptedException {
        Thread worker = new Thread(() -> {
            while (running) { /* spin */ }
            System.out.println("stopped cleanly");
        });
        worker.start();
        Thread.sleep(100);
        running = false; // guaranteed visible to worker
    }
}

Double-Checked Locking — volatile is critical

public class Singleton {
    // Without volatile, a partially-constructed object can be seen by other threads
    private static volatile Singleton instance;

    private Singleton() {}

    public static Singleton getInstance() {
        if (instance == null) {                      // first check (no lock)
            synchronized (Singleton.class) {
                if (instance == null) {              // second check (with lock)
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}

// Better: Initialization-On-Demand Holder (no volatile needed)
public class BetterSingleton {
    private BetterSingleton() {}

    private static class Holder {
        static final BetterSingleton INSTANCE = new BetterSingleton();
    }

    public static BetterSingleton getInstance() { return Holder.INSTANCE; }
}

2.4 synchronized — Intrinsic Locks

Every Java object has a built-in monitor lock. synchronized acquires that lock on entry and releases it on exit — even if an exception is thrown.

public class Counter {
    private int count = 0;

    // Locks on 'this' — entire method is critical section
    public synchronized void increment() {
        count++;
    }

    // Locking on a dedicated lock object — finer grained
    private final Object lock = new Object();
    private int value = 0;

    public void add(int n) {
        synchronized (lock) {
            value += n;
        }
    }
}

// Reentrancy — a thread can re-acquire its own lock
public class Reentrant {
    public synchronized void outer() {
        System.out.println("outer");
        inner(); // same thread re-acquires lock — no deadlock
    }
    public synchronized void inner() {
        System.out.println("inner");
    }
}

wait / notify — Thread Communication on a Monitor

public class ProducerConsumer {
    private final Queue<Integer> queue = new LinkedList<>();
    private final int CAPACITY = 5;

    public synchronized void produce(int val) throws InterruptedException {
        while (queue.size() == CAPACITY) {
            wait();                          // releases lock, waits
        }
        queue.add(val);
        System.out.println("produced " + val);
        notifyAll();                         // wake up waiting consumers
    }

    public synchronized int consume() throws InterruptedException {
        while (queue.isEmpty()) {
            wait();
        }
        int val = queue.poll();
        notifyAll();                         // wake up waiting producers
        return val;
    }
}
Always use while (not if) around wait(). Spurious wakeups are allowed by the JVM spec — the condition must be re-checked every time the thread wakes.
Prefer notifyAll() over notify(). notify() wakes only one random thread, which may not be the one that can actually proceed.
Phase 3

Atomic Operations — Lock-Free Building Blocks

The java.util.concurrent.atomic package provides classes that support lock-free, thread-safe operations on single variables. They use CPU-level compare-and-swap (CAS) instructions — no locks, no blocking, maximum throughput.

3.1 Why Atomics?

Consider count++. This is actually three operations: read, increment, write. Between the read and the write, another thread can swoop in. This is a read-modify-write race condition.

// BROKEN — race condition
public class BrokenCounter {
    private int count = 0;

    public void increment() { count++; } // NOT atomic!
}

// FIXED — atomic
import java.util.concurrent.atomic.AtomicInteger;

public class AtomicCounter {
    private final AtomicInteger count = new AtomicInteger(0);

    public void  increment()         { count.incrementAndGet(); }
    public int   decrement()         { return count.decrementAndGet(); }
    public int   get()               { return count.get(); }
    public int   addAndGet(int n)   { return count.addAndGet(n); }
    public int   getAndAdd(int n)   { return count.getAndAdd(n); }
}

3.2 Compare-And-Swap (CAS) — How It Works

CAS is a single CPU instruction (CMPXCHG on x86) that atomically does: "if the current value equals expected, replace it with newValue, and tell me if it succeeded."

// CAS in pseudocode (what the CPU does atomically)
boolean compareAndSet(int expected, int newVal) {
    if (current == expected) {
        current = newVal;
        return true;
    }
    return false;
}

// Manual CAS loop (how atomics are implemented internally)
public int safeIncrement(AtomicInteger ai) {
    int oldVal, newVal;
    do {
        oldVal = ai.get();
        newVal = oldVal + 1;
    } while (!ai.compareAndSet(oldVal, newVal));
    // Only exits when CAS succeeds — no lock needed
    return newVal;
}

3.3 AtomicReference — Lock-Free Pointer Swap

import java.util.concurrent.atomic.AtomicReference;

record Config(String host, int port) {}

public class ConfigHolder {
    private final AtomicReference<Config> ref =
        new AtomicReference<>(new Config("localhost", 8080));

    // Any thread can hot-swap the config atomically
    public void updateConfig(Config newCfg) {
        ref.set(newCfg);
    }

    public Config getConfig() { return ref.get(); }
}

3.4 The ABA Problem

CAS checks if the value is still A before writing B. But what if another thread changed A → C → A in between? CAS sees A and thinks nothing changed — but the state may be completely different. This is the ABA problem.

// Fix: use AtomicStampedReference — adds a version/stamp counter
import java.util.concurrent.atomic.AtomicStampedReference;

public class ABAFix {
    private final AtomicStampedReference<String> ref =
        new AtomicStampedReference<>("A", 0);

    public boolean safeUpdate(String expected, String newVal) {
        int[] stampHolder = new int[1];
        String current = ref.get(stampHolder);
        int stamp = stampHolder[0];

        if (!current.equals(expected)) return false;
        // CAS on both value AND stamp — ABA now impossible
        return ref.compareAndSet(expected, newVal, stamp, stamp + 1);
    }
}

3.5 LongAdder — High-Contention Counters

Under high contention, many threads spinning on the same AtomicLong cause a lot of CAS failures and retries. LongAdder solves this by maintaining an array of cells — each thread updates its own cell, and the total is summed on sum().

import java.util.concurrent.atomic.*;

public class ContendedCounter {
    private final LongAdder  adder  = new LongAdder();
    private final AtomicLong atomic = new AtomicLong();

    public void incrementAdder()  { adder.increment(); }
    public long sumAdder()        { return adder.sum(); }    // sums all cells

    public void incrementAtomic() { atomic.incrementAndGet(); }
    public long getAtomic()       { return atomic.get(); }
}
// Rule: LongAdder for high-contention write-heavy counters
// AtomicLong when you need compareAndSet or exact read-then-write

Phase 4

Explicit Locks — More Power Than synchronized

synchronized is convenient but limited: you can't try to acquire a lock without blocking, you can't be interrupted while waiting, and you can't have separate wait-sets. The java.util.concurrent.locks package fixes all of this.

4.1 ReentrantLock

ReentrantLock is a drop-in replacement for synchronized with superpowers. It is re-entrant — a thread that holds it can acquire it again without deadlocking.

import java.util.concurrent.locks.ReentrantLock;

public class SafeAccount {
    private double balance;
    private final ReentrantLock lock = new ReentrantLock();

    public void deposit(double amount) {
        lock.lock();
        try {
            balance += amount;
        } finally {
            lock.unlock();   // ALWAYS in finally — never skip this
        }
    }

    // tryLock — non-blocking attempt
    public boolean tryDeposit(double amount) {
        if (lock.tryLock()) {
            try {
                balance += amount;
                return true;
            } finally {
                lock.unlock();
            }
        }
        return false; // could not acquire lock right now
    }

    // tryLock with timeout
    public boolean timedDeposit(double amount) throws InterruptedException {
        if (lock.tryLock(500, java.util.concurrent.TimeUnit.MILLISECONDS)) {
            try {
                balance += amount;
                return true;
            } finally {
                lock.unlock();
            }
        }
        return false;
    }
}

Fair vs Non-Fair Lock

ReentrantLock fairLock    = new ReentrantLock(true);  // FIFO order
ReentrantLock unfairLock  = new ReentrantLock(false); // default, faster
Fair locks prevent starvation but have lower throughput (roughly 2–3× slower). Use only when starvation is an actual measured problem.

4.2 Condition — Multiple Wait Sets

A Condition is like wait/notify but attached to a specific ReentrantLock. You can have multiple conditions on one lock, solving the "wrong thread wakes up" problem of notifyAll().

import java.util.concurrent.locks.*;

public class BoundedBuffer<T> {
    private final Object[] items;
    private int head, tail, count;

    private final ReentrantLock lock = new ReentrantLock();
    private final Condition notFull  = lock.newCondition(); // producers wait here
    private final Condition notEmpty = lock.newCondition(); // consumers wait here

    public BoundedBuffer(int capacity) { items = new Object[capacity]; }

    public void put(T item) throws InterruptedException {
        lock.lock();
        try {
            while (count == items.length) notFull.await(); // only producers sleep here
            items[tail] = item;
            tail = (tail + 1) % items.length;
            count++;
            notEmpty.signal(); // wake ONE consumer, not all threads
        } finally { lock.unlock(); }
    }

    @SuppressWarnings("unchecked")
    public T take() throws InterruptedException {
        lock.lock();
        try {
            while (count == 0) notEmpty.await();   // only consumers sleep here
            T item = (T) items[head];
            head = (head + 1) % items.length;
            count--;
            notFull.signal(); // wake ONE producer
            return item;
        } finally { lock.unlock(); }
    }
}
This is precisely how ArrayBlockingQueue is implemented in the JDK. Two conditions = producer wakes only consumers, consumer wakes only producers. Much more efficient than notifyAll().

4.3 ReadWriteLock

If data is read far more than it is written, a read-write lock lets multiple readers proceed simultaneously while writers get exclusive access.

import java.util.concurrent.locks.*;

public class SharedCache {
    private final Map<String, String> data = new HashMap<>();
    private final ReadWriteLock rwLock = new ReentrantReadWriteLock();
    private final Lock readLock  = rwLock.readLock();
    private final Lock writeLock = rwLock.writeLock();

    public String get(String key) {
        readLock.lock();
        try { return data.get(key); }
        finally { readLock.unlock(); }
    }

    public void put(String key, String val) {
        writeLock.lock();
        try { data.put(key, val); }
        finally { writeLock.unlock(); }
    }

    // Lock DOWNGRADE: write → read (allowed)
    public String putAndGet(String key, String val) {
        writeLock.lock();
        try {
            data.put(key, val);
            readLock.lock();   // acquire read lock while holding write lock
        } finally {
            writeLock.unlock(); // release write, still holding read
        }
        try { return data.get(key); }
        finally { readLock.unlock(); }
    }
}
Lock upgrading (read → write) is NOT supported by ReentrantReadWriteLock and will deadlock.

4.4 StampedLock — Optimistic Reads

StampedLock (Java 8+) has three modes: write, read, and optimistic read. Optimistic read is lock-free — you read, then validate that no writer snuck in. If a write happened, fall back to a full read lock.

import java.util.concurrent.locks.StampedLock;

public class Point {
    private double x, y;
    private final StampedLock sl = new StampedLock();

    public void move(double dx, double dy) {
        long stamp = sl.writeLock();
        try { x += dx; y += dy; }
        finally { sl.unlockWrite(stamp); }
    }

    public double distanceFromOrigin() {
        long stamp = sl.tryOptimisticRead();   // no lock taken!
        double cx = x, cy = y;                  // read fields

        if (!sl.validate(stamp)) {              // was there a write in between?
            stamp = sl.readLock();             // fall back to real read lock
            try { cx = x; cy = y; }
            finally { sl.unlockRead(stamp); }
        }
        return Math.sqrt(cx * cx + cy * cy);
    }
}
StampedLock is NOT reentrant. If a thread tries to acquire it twice, it deadlocks. Also, it does not support Condition.

Phase 5

Thread Pools & Executors

Never create raw threads in production code. Thread creation is expensive (~1 ms, ~1 MB stack). Thread pools reuse threads, control parallelism, and provide lifecycle management. The entire java.util.concurrent executor framework is built on this idea.

5.1 ExecutorService — Factory Methods

import java.util.concurrent.*;

// Fixed pool — good for CPU-bound tasks, size = # cores
ExecutorService fixed = Executors.newFixedThreadPool(
    Runtime.getRuntime().availableProcessors()
);

// Cached pool — grows unbounded, idle threads die after 60s
// DANGEROUS for bursty workloads — can create thousands of threads
ExecutorService cached = Executors.newCachedThreadPool();

// Single thread — guaranteed serial execution, replaces lost threads
ExecutorService single = Executors.newSingleThreadExecutor();

// Scheduled — run after delay or periodically
ScheduledExecutorService sched = Executors.newScheduledThreadPool(2);
sched.scheduleAtFixedRate(() -> heartbeat(), 0, 1, TimeUnit.SECONDS);

// Work-stealing — uses ForkJoinPool internally, parallelism = # cores
ExecutorService stealing = Executors.newWorkStealingPool();

// Virtual thread per task — Java 21+ (the new default for I/O-bound)
ExecutorService vt = Executors.newVirtualThreadPerTaskExecutor();

Proper Shutdown

ExecutorService exec = Executors.newFixedThreadPool(4);
try {
    exec.submit(() -> doWork());
} finally {
    exec.shutdown();                              // stop accepting new tasks
    if (!exec.awaitTermination(30, TimeUnit.SECONDS)) {
        exec.shutdownNow();                        // interrupt running tasks
    }
}
shutdown() is graceful — running and queued tasks finish. shutdownNow() interrupts running tasks and returns queued ones. A thread pool that is never shut down leaks threads in a web container.

5.2 ThreadPoolExecutor — The Real Class

Every factory method in Executors returns a configured ThreadPoolExecutor. Understanding its 7-parameter constructor lets you tune it precisely.

import java.util.concurrent.*;

ThreadPoolExecutor pool = new ThreadPoolExecutor(
    4,                              // corePoolSize — always-alive threads
    8,                              // maximumPoolSize — max under load
    60L, TimeUnit.SECONDS,          // keepAliveTime — idle extra threads die
    new LinkedBlockingQueue<>(100), // workQueue — bounded!
    new ThreadFactory() {           // custom thread factory
        private final AtomicInteger n = new AtomicInteger();
        public Thread newThread(Runnable r) {
            Thread t = new Thread(r, "worker-" + n.incrementAndGet());
            t.setDaemon(false);
            return t;
        }
    },
    new ThreadPoolExecutor.CallerRunsPolicy() // rejection policy
);

5.3 Future and Callable

import java.util.concurrent.*;
import java.util.*;

public class FutureDemo {
    public static void main(String[] args) throws Exception {
        ExecutorService exec = Executors.newFixedThreadPool(3);

        // Submit returns a Future
        Future<Integer> f1 = exec.submit(() -> compute(10));
        Future<Integer> f2 = exec.submit(() -> compute(20));

        // Do other work while tasks run concurrently…
        System.out.println("tasks submitted");

        // get() blocks until result is ready
        System.out.println("sum = " + (f1.get() + f2.get()));

        // invokeAll — submit many, wait for ALL
        List<Callable<String>> tasks = List.of(
            () -> fetch("url1"),
            () -> fetch("url2"),
            () -> fetch("url3")
        );
        List<Future<String>> results = exec.invokeAll(tasks);
        results.forEach(f -> {
            try { System.out.println(f.get()); }
            catch (Exception e) { e.printStackTrace(); }
        });

        // invokeAny — submit many, return FIRST successful result
        String fastest = exec.invokeAny(tasks); // cancels the rest
        System.out.println("fastest: " + fastest);

        exec.shutdown();
    }

    static int compute(int n) { return n * n; }
    static String fetch(String url) { return "data from " + url; }
}
future.get() wraps task exceptions in ExecutionException. Always unwrap with getCause() to get the real error.
Phase 6

Synchronization Utilities — The Coordination Toolkit

When threads need to coordinate — wait for each other, take turns, or limit access to a resource — the utilities in this chapter are your tools. Each solves a distinct coordination pattern.

6.1 CountDownLatch — One-Shot Gate

Initialize with a count. Any thread can call countDown() to decrement. Threads waiting on await() are released when the count reaches zero. It cannot be reset — use CyclicBarrier if you need that.

import java.util.concurrent.*;

public class StartGun {
    public static void main(String[] args) throws InterruptedException {
        int RUNNERS = 5;
        CountDownLatch ready  = new CountDownLatch(RUNNERS); // all racers ready
        CountDownLatch go     = new CountDownLatch(1);       // start signal
        CountDownLatch finish = new CountDownLatch(RUNNERS); // all finished

        for (int i = 0; i < RUNNERS; i++) {
            final int id = i;
            Thread.ofVirtual().start(() -> {
                try {
                    System.out.println("Runner " + id + " ready");
                    ready.countDown();   // signal I'm ready
                    go.await();         // wait for start gun
                    System.out.println("Runner " + id + " running!");
                    finish.countDown(); // signal I'm done
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                }
            });
        }

        ready.await();            // wait for all runners to be ready
        System.out.println("GO!");
        go.countDown();           // fire the start gun
        finish.await();           // wait for all to finish
        System.out.println("Race over!");
    }
}

6.2 CyclicBarrier — Repeating Checkpoint

All threads wait at the barrier. When the last one arrives, a barrier action (optional) runs, then all threads are released. The barrier resets — threads can use it for the next round.

import java.util.concurrent.*;

public class ParallelPhases {
    static final int THREADS = 3;
    static final int ROUNDS  = 3;

    public static void main(String[] args) {
        CyclicBarrier barrier = new CyclicBarrier(
            THREADS,
            () -> System.out.println("--- All threads reached checkpoint ---")
        );

        for (int i = 0; i < THREADS; i++) {
            final int id = i;
            new Thread(() -> {
                for (int r = 0; r < ROUNDS; r++) {
                    try {
                        doPhaseWork(id, r);
                        barrier.await(); // wait for all to finish this phase
                    } catch (Exception e) { Thread.currentThread().interrupt(); }
                }
            }).start();
        }
    }

    static void doPhaseWork(int id, int r) throws InterruptedException {
        Thread.sleep((long)(Math.random() * 300));
        System.out.printf("Thread %d finished round %d%n", id, r);
    }
}

6.3 Semaphore — Permit-Based Access Control

A semaphore maintains a set of permits. acquire() takes one (blocks if none available). release() gives one back. Useful for connection pools, rate limiting, and bounding concurrency.

import java.util.concurrent.*;

public class ConnectionPool {
    private final Semaphore permits;
    private final BlockingQueue<String> pool;

    public ConnectionPool(int size) {
        permits = new Semaphore(size, true); // fair
        pool    = new LinkedBlockingQueue<>();
        for (int i = 0; i < size; i++) pool.add("conn-" + i);
    }

    public String acquire() throws InterruptedException {
        permits.acquire();   // blocks if no permits
        return pool.poll();
    }

    public void release(String conn) {
        pool.add(conn);
        permits.release();
    }
}

6.4 Phaser — Dynamic Participants

Phaser is a flexible combination of CountDownLatch and CyclicBarrier. Participants can register and deregister dynamically. It advances through numbered phases.

import java.util.concurrent.*;

public class PhaserExample {
    public static void main(String[] args) {
        Phaser phaser = new Phaser(1); // register main thread as participant

        for (int i = 0; i < 3; i++) {
            phaser.register(); // register before starting thread
            final int id = i;
            new Thread(() -> {
                for (int phase = 0; phase < 3; phase++) {
                    System.out.printf("Thread %d, phase %d%n", id, phaser.getPhase());
                    phaser.arriveAndAwaitAdvance(); // arrive and wait for all
                }
                phaser.arriveAndDeregister(); // leave the phaser
            }).start();
        }

        for (int phase = 0; phase < 3; phase++) {
            phaser.arriveAndAwaitAdvance(); // main thread participates too
            System.out.println("=== Phase " + phase + " complete ===");
        }
        phaser.arriveAndDeregister();
    }
}

Phase 7

Concurrent Collections

Never wrap a non-thread-safe collection with Collections.synchronizedMap() if you can use a concurrent collection instead. The concurrent classes in java.util.concurrent are far more scalable, and they avoid the pitfall of synchronized wrappers requiring external locking for compound operations.

7.1 ConcurrentHashMap

In Java 8+, ConcurrentHashMap uses CAS + synchronized on individual bins (not segments like Java 7). Reads are non-blocking. Writes lock only a single bin, not the whole map. This gives near-linear scalability with thread count.

import java.util.concurrent.*;

public class WordCounter {
    private final ConcurrentHashMap<String, Integer> counts
        = new ConcurrentHashMap<>();

    public void count(String word) {
        // merge: atomically increment, or initialize to 1
        counts.merge(word, 1, Integer::sum);
    }

    // computeIfAbsent — create value only if key absent (great for maps of lists)
    public void addToGroup(String group, String item) {
        ConcurrentHashMap<String, List<String>> groups = new ConcurrentHashMap<>();
        groups.computeIfAbsent(group, k -> new CopyOnWriteArrayList<>()).add(item);
    }

    // Parallel bulk operations — Java 8+
    public void parallelOps() {
        long parallelismThreshold = 1; // 1 = use all available cores

        // forEach in parallel
        counts.forEach(parallelismThreshold,
            (k, v) -> System.out.println(k + "=" + v));

        // reduce to find most common word
        Integer max = counts.reduceValues(parallelismThreshold,
            (Integer a, Integer b) -> a > b ? a : b);
        System.out.println("max count: " + max);
    }
}
size() on ConcurrentHashMap is only approximate under concurrent modifications. Use mappingCount() (returns long) or treat size as a hint, not a guarantee.
Even though individual operations are atomic, compound operations like check-then-act are NOT. Use computeIfAbsent, compute, merge, or putIfAbsent for atomic compound ops.

public class PipelineDemo {

7.3 CopyOnWriteArrayList

Every write operation copies the entire underlying array. Reads are always non-blocking and iterate over a snapshot — ConcurrentModificationException is impossible.

import java.util.concurrent.*;

CopyOnWriteArrayList<String> listeners = new CopyOnWriteArrayList<>();

// Writers: infrequent adds/removes
listeners.add("listener1");
listeners.remove("listener1");

// Readers: many concurrent reads, iterate freely with no locking
for (String listener : listeners) {
    notifyListener(listener); // safe to iterate while others modify
}
Never use CopyOnWriteArrayList for large lists with frequent writes. Each write allocates and copies the entire array — O(n) per write.

Phase 8

Async Programming — CompletableFuture & Flow

Future is limiting — you can only block on get(). CompletableFuture (Java 8+) lets you compose asynchronous operations like a pipeline: transform, combine, handle errors — all without blocking a thread.

8.1 CompletableFuture Basics

import java.util.concurrent.*;

// Start async computation
CompletableFuture<String> cf = CompletableFuture.supplyAsync(() -> {
    simulateNetworkCall(); // runs in ForkJoinPool.commonPool()
    return "data";
});

// Chain transformations (runs in same thread or pool thread)
CompletableFuture<Integer> length = cf
    .thenApply(String::toUpperCase)   // transform result
    .thenApply(String::length);        // transform again

// thenAccept — consume result (void)
cf.thenAccept(result -> System.out.println("Got: " + result));

// thenRun — run action after completion (ignores result)
cf.thenRun(() -> System.out.println("pipeline done"));

Using a Custom Executor

ExecutorService ioPool = Executors.newVirtualThreadPerTaskExecutor();

// Always pass your own executor for I/O tasks.
// ForkJoinPool.commonPool() is for CPU-bound work only!
CompletableFuture<String> ioTask = CompletableFuture.supplyAsync(
    () -> readFromDatabase(),
    ioPool   // runs on virtual threads — scales to millions of I/O ops
);

8.2 Combining Futures

// thenCompose — flatMap: future returning a future (avoid nesting)
CompletableFuture<String> result =
    CompletableFuture.supplyAsync(() -> getUserId())
        .thenCompose(userId -> fetchUserAsync(userId));   // returns CF<User>

// thenCombine — merge two independent futures
CompletableFuture<String> priceF   = CompletableFuture.supplyAsync(() -> getPrice());
CompletableFuture<String> stockF   = CompletableFuture.supplyAsync(() -> getStock());
CompletableFuture<String> combined = priceF.thenCombine(stockF,
    (price, stock) -> price + " | stock: " + stock);

// allOf — wait for ALL futures (returns CF<Void>)
CompletableFuture<Void> all = CompletableFuture.allOf(priceF, stockF);
all.thenRun(() -> System.out.println("both done"));

// anyOf — complete when FIRST finishes (returns CF<Object>)
CompletableFuture<Object> fastest = CompletableFuture.anyOf(priceF, stockF);

8.3 Error Handling

CompletableFuture<String> safe = CompletableFuture.supplyAsync(() -> {
    if (Math.random() < 0.5) throw new RuntimeException("boom");
    return "ok";
})
.exceptionally(ex -> {
    // called only on exception — provides fallback value
    System.out.println("error: " + ex.getMessage());
    return "fallback";
})
.handle((result, ex) -> {
    // called always — either result or ex is non-null
    if (ex != null) return "handled error";
    return result.toUpperCase();
})
.whenComplete((result, ex) -> {
    // called always, cannot transform — good for logging/cleanup
    System.out.println("done: result=" + result + " ex=" + ex);
});

8.4 Real Pipeline Example

public class OrderPipeline {
    private final ExecutorService pool =
        Executors.newVirtualThreadPerTaskExecutor();

    public CompletableFuture<String> processOrder(int orderId) {
        return CompletableFuture
            .supplyAsync(() -> validateOrder(orderId), pool)
            .thenComposeAsync(order -> chargePaymentAsync(order), pool)
            .thenComposeAsync(payment -> fulfillOrderAsync(payment), pool)
            .thenApplyAsync(fulfillment -> sendConfirmation(fulfillment), pool)
            .exceptionally(ex -> "Order failed: " + ex.getCause().getMessage());
    }

    public static void main(String[] args) throws Exception {
        OrderPipeline pipeline = new OrderPipeline();

        // Process 100 orders concurrently
        List<CompletableFuture<String>> orders = IntStream.range(1, 101)
            .mapToObj(pipeline::processOrder)
            .toList();

        CompletableFuture.allOf(orders.toArray(new CompletableFuture[0])).join();
        orders.forEach(f -> System.out.println(f.join()));
    }

    private String  validateOrder(int id)   { return "order-" + id; }
    private CompletableFuture<String> chargePaymentAsync(String o) {
        return CompletableFuture.supplyAsync(() -> o + ":paid", pool);
    }
    private CompletableFuture<String> fulfillOrderAsync(String p) {
        return CompletableFuture.supplyAsync(() -> p + ":shipped", pool);
    }
    private String  sendConfirmation(String f)  { return f + ":confirmed"; }
}

8.5 Flow API — Reactive Streams (Java 9+)

The java.util.concurrent.Flow API defines four interfaces for reactive programming. The key feature is backpressure — the subscriber controls how fast the publisher sends data.

import java.util.concurrent.*;

public class FlowDemo {
    public static void main(String[] args) throws InterruptedException {
        // SubmissionPublisher is the built-in publisher
        try (SubmissionPublisher<String> publisher = new SubmissionPublisher<>()) {

            publisher.subscribe(new Flow.Subscriber<String>() {
                Flow.Subscription subscription;

                @Override public void onSubscribe(Flow.Subscription s) {
                    subscription = s;
                    s.request(1); // ask for 1 item at a time (backpressure)
                }

                @Override public void onNext(String item) {
                    System.out.println("received: " + item);
                    subscription.request(1); // ask for next
                }

                @Override public void onError(Throwable t) { t.printStackTrace(); }
                @Override public void onComplete() { System.out.println("done"); }
            });

            publisher.submit("item1");
            publisher.submit("item2");
            publisher.submit("item3");
            Thread.sleep(500); // let subscriber process
        } // auto-close calls onComplete
    }
}
Phase 9

Fork/Join Framework — Divide and Conquer

The Fork/Join framework (Java 7+) is designed for parallelizing recursive, divide-and-conquer algorithms. Its secret weapon is work stealing — idle threads steal tasks from busy threads' queues, keeping all cores saturated.

9.1 Work Stealing

Each worker thread in a ForkJoinPool has its own double-ended task queue (deque). When a thread runs out of tasks, it steals from the tail of another thread's deque. The original thread works from its own head. This minimizes contention and maximizes CPU utilization.

ForkJoinPool

import java.util.concurrent.*;

// The common pool — shared across JVM, size = # CPUs - 1
ForkJoinPool common = ForkJoinPool.commonPool();

// Custom pool — use when you need isolation from commonPool
// (important: don't block commonPool threads with I/O)
ForkJoinPool custom = new ForkJoinPool(
    Runtime.getRuntime().availableProcessors()
);

try {
    Long result = custom.invoke(new SumTask(0, 1_000_000));
    System.out.println("Sum = " + result);
} finally {
    custom.shutdown();
}
Never do blocking I/O in ForkJoinPool.commonPool(). It has a fixed number of threads equal to CPU count. Blocking there starves other tasks using the pool (including parallel streams!).

9.2 RecursiveTask — Returns a Value

import java.util.concurrent.*;

public class SumTask extends RecursiveTask<Long> {
    private static final int THRESHOLD = 10_000; // tune based on workload
    private final long[] arr;
    private final int  lo, hi;

    public SumTask(long[] arr, int lo, int hi) {
        this.arr = arr; this.lo = lo; this.hi = hi;
    }

    @Override
    protected Long compute() {
        int size = hi - lo;

        if (size <= THRESHOLD) {          // base case: compute directly
            long sum = 0;
            for (int i = lo; i < hi; i++) sum += arr[i];
            return sum;
        }

        int mid = lo + size / 2;
        SumTask left  = new SumTask(arr, lo,  mid); // left half
        SumTask right = new SumTask(arr, mid, hi);  // right half

        left.fork();           // submit left asynchronously
        long rightResult = right.compute(); // compute right in current thread
        long leftResult  = left.join();   // wait for left

        return leftResult + rightResult;
    }

    public static void main(String[] args) {
        int N = 1_000_000;
        long[] arr = new long[N];
        for (int i = 0; i < N; i++) arr[i] = i + 1;

        long sum = ForkJoinPool.commonPool().invoke(new SumTask(arr, 0, N));
        System.out.println("Sum = " + sum); // 500000500000
    }
}

9.3 RecursiveAction — No Return Value

public class ParallelSort extends RecursiveAction {
    private static final int THRESHOLD = 1000;
    private final int[] arr;
    private final int  lo, hi;

    public ParallelSort(int[] arr, int lo, int hi) {
        this.arr = arr; this.lo = lo; this.hi = hi;
    }

    @Override
    protected void compute() {
        if (hi - lo < THRESHOLD) {
            java.util.Arrays.sort(arr, lo, hi);
            return;
        }
        int mid = (lo + hi) / 2;
        RecursiveAction left  = new ParallelSort(arr, lo,  mid);
        RecursiveAction right = new ParallelSort(arr, mid, hi);
        invokeAll(left, right);            // fork both, wait for both
        merge(arr, lo, mid, hi);
    }

    private void merge(int[] a, int lo, int mid, int hi) {
        int[] tmp = java.util.Arrays.copyOfRange(a, lo, hi);
        int i = 0, j = mid - lo, k = lo;
        while (i < mid - lo && j < hi - lo)
            a[k++] = tmp[i] <= tmp[j] ? tmp[i++] : tmp[j++];
        while (i < mid - lo) a[k++] = tmp[i++];
        while (j < hi - lo)  a[k++] = tmp[j++];
    }
}
Pattern: always fork() one subtask and compute() the other directly. Never fork both and join both — that wastes a thread slot.
Phase 10

ThreadLocal & ScopedValue

Sometimes you need per-thread state — a database connection, a user session ID, a formatter. ThreadLocal is the classic tool. ScopedValue (stable in Java 21+, refined in Java 25) is the modern replacement that works correctly with virtual threads.

10.1 ThreadLocal

Each thread that accesses a ThreadLocal variable gets its own isolated copy. Threads do not see each other's values.

public class RequestContext {
    // Each thread gets its own String (user ID)
    private static final ThreadLocal<String> userId =
        ThreadLocal.withInitial(() -> "anonymous");

    public static void setUser(String id) { userId.set(id);     }
    public static String getUser()       { return userId.get(); }
    public static void  clear()          { userId.remove();    } // CRITICAL!
}

// In a web framework — set at request start, clear at request end
public class RequestFilter {
    public void doFilter(Request req) {
        try {
            RequestContext.setUser(req.getHeader("X-User-Id"));
            processRequest(req);
        } finally {
            RequestContext.clear(); // ALWAYS clear in finally — prevents leaks
        }
    }
}
ThreadLocal Memory Leak in Thread Pools: Thread pool threads are reused. If you call set() but never remove(), the value stays in the thread's map forever — preventing garbage collection of the value AND leaking state from one request to the next.

InheritableThreadLocal — Child Inherits Parent's Value

private static final InheritableThreadLocal<String> traceId =
    new InheritableThreadLocal<>();

// Parent thread sets it
traceId.set("trace-xyz");

// Child thread inherits the value at creation time
new Thread(() -> System.out.println(traceId.get())).start(); // prints "trace-xyz"
InheritableThreadLocal does NOT work reliably with thread pools — the child thread is the pool thread created at startup, not at task submission time.

10.2 ScopedValue — The Modern Way (Java 21+)

ScopedValue is immutable within a scope — once bound, it cannot be changed. It works correctly with virtual threads and structured concurrency. The value is automatically unbound when the scope exits.

import java.lang.ScopedValue;

public class ScopedValueDemo {
    static final ScopedValue<String> USER = ScopedValue.newInstance();
    static final ScopedValue<String> TRACE_ID = ScopedValue.newInstance();

        ScopedValue.where(USER, "maruf")
                   .where(TRACE_ID, "trace-001")
                   .run(() -> {
                       handleRequest();  // USER and TRACE_ID visible here

        System.out.println("User: "    + USER.get());
        System.out.println("Trace: "   + TRACE_ID.get());
        callService();

        // ScopedValue flows down the call stack — no need to pass as parameter
        System.out.println("service sees user: " + USER.get());

Phase 11

Virtual Threads & Structured Concurrency

Project Loom (stable since Java 21) is the biggest change to Java concurrency in 20 years. Virtual threads let you write simple blocking code that scales to millions of concurrent operations. Structured concurrency gives async code the reliability of synchronous code.

11.1 The Problem with Platform Threads

A platform thread consumes ~1 MB of native stack. You can have maybe 10,000 before you run out of memory. Modern applications need to handle hundreds of thousands of concurrent connections. The old solution was async/reactive programming — callbacks, CompletableFuture chains. It works but it's hard to read, debug, and reason about.

11.2 Virtual Threads

A virtual thread is a lightweight thread managed by the JVM, not the OS. It is multiplexed onto carrier threads (a pool of platform threads). When a virtual thread blocks (on I/O, sleep, lock), it is unmounted from its carrier — the carrier is free to run another virtual thread. No OS thread is blocked.

import java.util.concurrent.*;

public class VirtualThreadDemo {
    public static void main(String[] args) throws Exception {

        // 1. Single virtual thread
        Thread vt = Thread.ofVirtual()
            .name("vt-1")
            .start(() -> System.out.println("hello from virtual thread"));
        vt.join();

        // 2. Virtual thread executor — one virtual thread per task (idiomatic)
        try (ExecutorService exec = Executors.newVirtualThreadPerTaskExecutor()) {
            IntStream.range(0, 10_000).forEach(i ->
                exec.submit(() -> {
                    simulateIoWait(); // blocks the virtual thread, not an OS thread
                    System.out.println("task " + i + " done");
                })
            );
        } // auto-close = shutdown + awaitTermination

        // 3. isVirtual — check if current thread is virtual
        Thread.ofVirtual().start(() ->
            System.out.println("virtual? " + Thread.currentThread().isVirtual())
        ).join();
    }

    static void simulateIoWait() {
        try { Thread.sleep(100); } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
    }
}

Pinning — The One Pitfall

A virtual thread is pinned to its carrier thread when:

  • It holds a synchronized monitor lock
  • It is inside a native method or foreign function

A pinned virtual thread blocks its carrier — defeating the whole point. The fix is to use ReentrantLock instead of synchronized.

// BAD — causes pinning
synchronized void badMethod() {
    doIoWork(); // virtual thread is pinned here — carrier thread blocked
}

// GOOD — no pinning
private final ReentrantLock lock = new ReentrantLock();
void goodMethod() {
    lock.lock();
    try {
        doIoWork(); // virtual thread unmounts cleanly, no pinning
    } finally { lock.unlock(); }
}
Java 25 (JEP in progress) is working on removing the synchronized pinning limitation. Until then, prefer ReentrantLock in virtual thread contexts.

11.3 Structured Concurrency (Java 21+ — stable Java 25)

Unstructured async code has a core problem: if you fork two tasks and one fails, the other keeps running, leaking resources and making debugging hell. Structured concurrency guarantees that all forked tasks finish (succeed, fail, or are cancelled) before the parent scope exits.

import java.util.concurrent.*;

public class StructuredDemo {
    record Order(String payment, String shipment) {}

    static Order fetchOrder(int orderId) throws Exception {
        // ShutdownOnFailure — if any subtask fails, others are cancelled
        try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
            StructuredTaskScope.Subtask<String> payment =
                scope.fork(() -> fetchPayment(orderId));  // runs in virtual thread

            StructuredTaskScope.Subtask<String> shipment =
                scope.fork(() -> fetchShipment(orderId)); // runs in virtual thread

            scope.join();           // wait for all subtasks
            scope.throwIfFailed(); // propagate exceptions

            return new Order(payment.get(), shipment.get());
        }
        // Scope closed: both subtasks are guaranteed done or cancelled here
    }

    // ShutdownOnSuccess — return first successful result, cancel the rest
    static String raceForFastest(int id) throws Exception {
        try (var scope = new StructuredTaskScope.ShutdownOnSuccess<String>()) {
            scope.fork(() -> fetchFromServer1(id));
            scope.fork(() -> fetchFromServer2(id));
            scope.fork(() -> fetchFromServer3(id));
            scope.join();
            return scope.result(); // returns first success, others cancelled
        }
    }

    static String fetchPayment(int id)  throws InterruptedException {
        Thread.sleep(100); return "paid-" + id;
    }
    static String fetchShipment(int id) throws InterruptedException {
        Thread.sleep(150); return "shipped-" + id;
    }
    static String fetchFromServer1(int id) throws InterruptedException {
        Thread.sleep(200); return "s1";
    }
    static String fetchFromServer2(int id) throws InterruptedException {
        Thread.sleep(100); return "s2";
    }
    static String fetchFromServer3(int id) throws InterruptedException {
        Thread.sleep(300); return "s3";
    }
}
The combination of virtual threads + structured concurrency + scoped values is the future of Java concurrency. You write straight-line blocking code, it scales like async, and the structure guarantees no resource leaks.

Failure Modes — What Breaks and Why

Concurrency bugs are the hardest bugs to find and fix. They are non-deterministic, often timing-dependent, and may only appear under load in production. This chapter shows you every major failure mode — with code that demonstrates the problem and the fix.

12.1 Deadlock

Four conditions for deadlock (all must be true simultaneously):

  1. Mutual exclusion — resource held exclusively
  2. Hold and wait — thread holds a lock while waiting for another
  3. No preemption — locks cannot be forcibly taken
  4. Circular wait — thread A waits for B, B waits for A
// Classic deadlock — two threads, two locks, opposite order
public class Deadlock {
    static final Object lockA = new Object();
    static final Object lockB = new Object();

    public static void main(String[] args) {
        new Thread(() -> {
            synchronized (lockA) {
                sleep(50);
                synchronized (lockB) { System.out.println("Thread 1 done"); }
            }
        }).start();

        new Thread(() -> {
            synchronized (lockB) {   // ← opposite order → deadlock!
                sleep(50);
                synchronized (lockA) { System.out.println("Thread 2 done"); }
            }
        }).start();
    }
}

// FIX 1: Consistent lock ordering — always lock A before B
synchronized (lockA) {
    synchronized (lockB) { /* both threads same order */ }
}

// FIX 2: tryLock with timeout — back off if can't acquire both
boolean success = false;
while (!success) {
    if (lockA.tryLock(100, TimeUnit.MILLISECONDS)) {
        try {
            if (lockB.tryLock(100, TimeUnit.MILLISECONDS)) {
                try { doWork(); success = true; }
                finally { lockB.unlock(); }
            }
        } finally { lockA.unlock(); }
    }
    if (!success) Thread.sleep(new Random().nextInt(50)); // random backoff
}

12.5 False Sharing

CPUs read/write memory in cache lines (~64 bytes). If two threads write to different variables that happen to share a cache line, they invalidate each other's cache constantly — even though they are working on different data. This is called false sharing.

// BROKEN — counter1 and counter2 are likely on the same cache line
public class FalseSharing {
    volatile long counter1 = 0;  // Thread 1 writes this
    volatile long counter2 = 0;  // Thread 2 writes this — false sharing!
}

// FIX: pad each counter to its own cache line (64 bytes)
public class PaddedCounter {
    volatile long p0, p1, p2, p3, p4, p5, p6; // 56 bytes padding before
    volatile long value = 0;                   // 8 bytes — own cache line
    volatile long q0, q1, q2, q3, q4, q5, q6; // 56 bytes padding after
}

// Better fix: @Contended annotation (requires JVM flag -XX:+EnableContendedPadding)
// Or in Java 9+ module-aware code: jdk.internal.vm.annotation.Contended
@jdk.internal.vm.annotation.Contended
public class ContendedCounter {
    volatile long value = 0;
}
False sharing is invisible in a thread dump. You can only detect it with performance profiling (JFR, async-profiler, Linux perf). It can cause 10–100x throughput degradation on shared counters.

13.1 Amdahl's Law — The Limit of Parallelism

If a fraction P of your program can be parallelized and (1-P) must run serially, the max speedup with N cores is:

Speedup = 1 / ((1-P) + P/N)

If 90% is parallel: with 10 cores → max 5.3× speedup. With 100 cores → max 9.2× speedup. The serial 10% dominates. This is why eliminating serial bottlenecks matters far more than adding cores.

Lock Striping

// Instead of one lock for the whole structure,
// use N locks — threads on different buckets don't contend
public class StripedMap<K, V> {
    private static final int STRIPES = 16;
    private final Object[]        locks = new Object[STRIPES];
    private final HashMap<K,V>[] maps;

    @SuppressWarnings("unchecked")
    public StripedMap() {
        maps = (HashMap<K,V>[]) new HashMap[STRIPES];
        for (int i = 0; i < STRIPES; i++) {
            locks[i] = new Object();
            maps[i]  = new HashMap<>();
        }
    }

    private int stripe(Object key) {
        return (System.identityHashCode(key) & 0x7fffffff) % STRIPES;
    }

    public V get(K key) {
        int s = stripe(key);
        synchronized (locks[s]) { return maps[s].get(key); }
    }

    public void put(K key, V val) {
        int s = stripe(key);
        synchronized (locks[s]) { maps[s].put(key, val); }
    }
}
// ConcurrentHashMap does exactly this internally

Concurrency patterns are proven solutions to recurring coordination problems. Understanding them lets you recognize the structure of a problem and apply a known-good solution.

Pattern 1: Immutable Object

An immutable object's state never changes after construction. No synchronization needed — ever.

public final class Money {
    private final long   amount;
    private final String currency;

    public Money(long amount, String currency) {
        this.amount   = amount;
        this.currency = currency;
    }

    public Money add(Money other) {
        if (!currency.equals(other.currency)) throw new IllegalArgumentException();
        return new Money(amount + other.amount, currency); // returns new object
    }
    // No setters. No mutable fields. Thread-safe by construction.
}

Pattern 2: Monitor Object

Encapsulate all shared state and the lock in one class. All public methods are synchronized. Internal state is private.

public class BoundedStack<T> {
    private final Object[] items;
    private int top = -1;

    public BoundedStack(int cap) { items = new Object[cap]; }

    public synchronized void push(T item) throws InterruptedException {
        while (top == items.length - 1) wait();
        items[++top] = item;
        notifyAll();
    }

    @SuppressWarnings("unchecked")
    public synchronized T pop() throws InterruptedException {
        while (top == -1) wait();
        T item = (T) items[top--];
        notifyAll();
        return item;
    }
}

Pattern 3: Worker Thread (Thread Pool)

public class WorkerPool {
    private final BlockingQueue<Runnable> taskQueue;
    private final List<Thread> workers = new ArrayList<>();

    public WorkerPool(int nThreads, int queueCap) {
        taskQueue = new LinkedBlockingQueue<>(queueCap);
        for (int i = 0; i < nThreads; i++) {
            Thread w = new Thread(() -> {
                while (!Thread.currentThread().isInterrupted()) {
                    try {
                        Runnable task = taskQueue.take(); // blocks until task available
                        task.run();
                    } catch (InterruptedException e) {
                        Thread.currentThread().interrupt();
                    }
                }
            }, "worker-" + i);
            workers.add(w);
            w.start();
        }
    }

    public void submit(Runnable task) throws InterruptedException {
        taskQueue.put(task); // blocks if queue full (backpressure)
    }

    public void shutdown() { workers.forEach(Thread::interrupt); }
}

Pattern 4: Active Object

Method calls are turned into queued async invocations. The caller gets a Future back immediately. This decouples method invocation from execution.

public class ActivePrinter {
    private final ExecutorService exec =
        Executors.newSingleThreadExecutor(); // one thread processes all calls

    // Caller gets a Future — doesn't block
    public Future<Void> print(String msg) {
        return exec.submit(() -> {
            System.out.println(Thread.currentThread().getName() + ": " + msg);
            return (Void) null;
        });
    }

    public void shutdown() { exec.shutdown(); }
}

public class LogPipeline {
    private static final String POISON = "__SHUTDOWN__";
    private final BlockingQueue<String> queue = new LinkedBlockingQueue<>(1000);

    public void log(String msg) {

        });
    }

    public void shutdown() {
        try { queue.put(POISON); }
        catch (InterruptedException e) { Thread.currentThread().interrupt(); }
    }
}

15.1 Semaphore-Based Connection Pool

import java.util.concurrent.*;

public class GenericPool<T> {
    private final BlockingQueue<T> pool;
    private final Semaphore       permits;

    public GenericPool(List<T> resources) {
        pool    = new LinkedBlockingQueue<>(resources);
        permits = new Semaphore(resources.size(), true);
    }

    public T borrow() throws InterruptedException {
        permits.acquire();
        return pool.poll(); // never null — permit guarantees availability
    }

    public T tryBorrow(long timeout, TimeUnit unit) throws InterruptedException {
        if (!permits.tryAcquire(timeout, unit)) return null;
        return pool.poll();
    }

    public void release(T resource) {
        pool.add(resource);
        permits.release();
    }

    // Usage
    public static void main(String[] args) throws Exception {
        GenericPool<String> connPool = new GenericPool<>(
            List.of("conn-1", "conn-2", "conn-3")
        );
        String conn = connPool.borrow();
        try {
            useConnection(conn);
        } finally {
            connPool.release(conn); // always release in finally
        }
    }

    static void useConnection(String c) { System.out.println("using " + c); }
}

15.2 Token Bucket Rate Limiter

import java.util.concurrent.*;
import java.util.concurrent.atomic.*;

public class RateLimiter {
    private final int             maxTokens;
    private final AtomicInteger   tokens;
    private final ScheduledExecutorService refiller;

    public RateLimiter(int tokensPerSecond) {
        this.maxTokens = tokensPerSecond;
        this.tokens    = new AtomicInteger(tokensPerSecond);
        this.refiller  = Executors.newSingleThreadScheduledExecutor(r -> {
            Thread t = new Thread(r, "rate-limiter-refiller");
            t.setDaemon(true);
            return t;
        });
        refiller.scheduleAtFixedRate(() ->
            tokens.set(maxTokens),       // refill bucket every second
            1, 1, TimeUnit.SECONDS
        );
    }

    public boolean tryAcquire() {
        while (true) {
            int current = tokens.get();
            if (current == 0) return false;
            if (tokens.compareAndSet(current, current - 1)) return true;
            // CAS failed — another thread took a token, retry
        }
    }

    public void shutdown() { refiller.shutdown(); }
}

15.3 Thread-Safe In-Memory Cache with Expiry

import java.util.concurrent.*;

public class ExpiringCache<K, V> {
    private record Entry<V>(V value, long expiresAt) {
        boolean isExpired() { return System.currentTimeMillis() > expiresAt; }
    }

    private final ConcurrentHashMap<K, Entry<V>> store = new ConcurrentHashMap<>();
    private final long             ttlMs;
    private final ScheduledExecutorService cleaner;

    public ExpiringCache(long ttlMs) {
        this.ttlMs = ttlMs;
        cleaner = Executors.newSingleThreadScheduledExecutor(r -> {
            Thread t = new Thread(r, "cache-cleaner");
            t.setDaemon(true);
            return t;
        });
        cleaner.scheduleAtFixedRate(
            () -> store.entrySet().removeIf(e -> e.getValue().isExpired()),
            ttlMs, ttlMs, TimeUnit.MILLISECONDS
        );
    }

    public void put(K key, V value) {
        store.put(key, new Entry<>(value, System.currentTimeMillis() + ttlMs));
    }

    public V get(K key) {
        Entry<V> entry = store.get(key);
        if (entry == null || entry.isExpired()) {
            store.remove(key);
            return null;
        }
        return entry.value;
    }

    public void shutdown() { cleaner.shutdown(); }
}

15.4 Pub/Sub Event Bus

import java.util.concurrent.*;
import java.util.function.*;

public class EventBus {
    // ConcurrentHashMap for topics, CopyOnWriteArrayList for listeners
    // — many concurrent publishes read listeners, rare subscribe/unsubscribe writes
    private final ConcurrentHashMap<String, CopyOnWriteArrayList<Consumer<Object>>>
        subscribers = new ConcurrentHashMap<>();

    private final ExecutorService dispatcher =
        Executors.newVirtualThreadPerTaskExecutor();

    @SuppressWarnings("unchecked")
    public <T> void subscribe(String topic, Consumer<T> listener) {
        subscribers
            .computeIfAbsent(topic, k -> new CopyOnWriteArrayList<>())
            .add((Consumer<Object>)(Consumer<?>) listener);
    }

    public void publish(String topic, Object event) {
        List<Consumer<Object>> listeners = subscribers.get(topic);
        if (listeners != null) {
            // Each listener runs in its own virtual thread — non-blocking publish
            listeners.forEach(l -> dispatcher.submit(() -> l.accept(event)));
        }
    }

    public void shutdown() { dispatcher.shutdown(); }

    public static void main(String[] args) throws InterruptedException {
        EventBus bus = new EventBus();

        bus.subscribe("order.placed", (String e) ->
            System.out.println("Email sent for: " + e));

        bus.subscribe("order.placed", (String e) ->
            System.out.println("Inventory updated for: " + e));

        bus.publish("order.placed", "order-42");
        Thread.sleep(100);
        bus.shutdown();
    }
}

15.5 Deadlock-Free Bank Transfer

public class BankAccount {
    private volatile double balance;
    private final int       id;
    private final ReentrantLock lock = new ReentrantLock();

    public BankAccount(int id, double balance) {
        this.id = id; this.balance = balance;
    }

    // Key: always lock the lower-id account first — consistent ordering prevents deadlock
    public static void transfer(BankAccount from, BankAccount to, double amount) {
        BankAccount first  = from.id < to.id ? from : to;
        BankAccount second = from.id < to.id ? to   : from;

        first.lock.lock();
        try {
            second.lock.lock();
            try {
                if (from.balance < amount) throw new IllegalStateException("NSF");
                from.balance -= amount;
                to.balance   += amount;
                System.out.printf("Transferred %.2f from %d to %d%n",
                    amount, from.id, to.id);
            } finally { second.lock.unlock(); }
        } finally { first.lock.unlock(); }
    }
}

The Golden Rules of Java Concurrency

  1. Prefer immutability. An object that never changes needs no synchronization.
  2. Confine mutable state to a single thread when possible.
  3. Always document thread-safety guarantees (or lack thereof).
  4. Never call an alien method (unknown code) while holding a lock.
  5. Hold locks for the minimum time necessary.
  6. Always acquire multiple locks in a consistent order.
  7. Always check conditions in a while loop after wait().
  8. Always call ThreadLocal.remove() when using thread pools.
  9. Never use synchronized inside a virtual thread hot path — use ReentrantLock.
  10. Use newVirtualThreadPerTaskExecutor() for I/O-bound work. Use ForkJoinPool for CPU-bound work.
  11. Profile before optimizing. Lock contention you cannot measure does not exist.
  12. Structure wins over cleverness. StructuredTaskScope > ad-hoc CompletableFuture chains.
